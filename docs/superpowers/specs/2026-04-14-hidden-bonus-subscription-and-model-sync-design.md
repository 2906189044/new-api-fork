# Hidden Bonus Subscription And Model Sync Design

## Goal

在不打乱现有套餐分组和自动路由的前提下，完成四件事：

1. 将 CLI API 中尚未同步的 Claude 系列模型和 Gemini 系列新上游模型补充到 New API 的模型广场与实际调用路由。
2. 核验既有品牌模型映射是否保持完好：
   - Gemini 3 / 3.1 系列仍映射到 GPT
   - Claude 4.5 系列仍映射到 GPT
3. 将“管理员专用 GPT 日限额计划”改造成只能后台开通、可与其他套餐叠加的附加权益订阅。
4. 在管理员“用户订阅管理”中，对该隐藏套餐增加历史已领提示，并禁止重复开通。

## Constraints

- 尽量不改动原项目既有套餐和分组逻辑。
- 公开套餐仍沿用现有 `UpgradeGroup` 行为。
- 隐藏套餐保持 `visible_to_user = false`，不在前台套餐页展示。
- 隐藏套餐的最新业务规则为：
  - 每日 `10 USD`
  - 每日刷新
  - 有效期 `10 天`
  - 仅限 GPT 系列
  - 免费用户和付费用户都可由管理员手动开通
  - 与其他套餐叠加时，GPT 系列优先扣该隐藏套餐额度
  - 其他模型继续从原套餐或钱包扣费
- 历史上开通过该隐藏套餐的用户，后续必须禁止再次开通。

## Current State

### 模型同步

- 模型广场来源于 `model.GetPricing()`，其底层依赖启用中的 `abilities + channels + models + ratio_setting`。
- 只往 `models` 表写元数据不够，若要实际调用，必须同步维护 `channels` 和 `abilities`。
- 当前 GPT/Claude/Gemini 已按 `gpt_core / claude_core / gemini_core / gpt52_unlimited` 分组，自动路由依赖这一结构，不能重排。

### 隐藏套餐

- 当前所有订阅都复用 `UpgradeGroup` 升降级逻辑。
- `CreateUserSubscriptionFromPlanTx` 会在创建订阅时直接更新用户 `group`。
- 这意味着隐藏套餐一旦配置了 `UpgradeGroup`，就会覆盖用户当前主分组，不能与其他套餐并行生效。

### 管理员后台

- 管理员开通用户订阅入口在 `UserSubscriptionsModal.jsx`。
- 目前仅能选择套餐并直接提交创建，没有针对“已领过隐藏福利”的前置提醒。
- 后端 `AdminCreateUserSubscription` 也没有对“历史已领取隐藏套餐”做防重校验。

## Approaches Considered

### 方案 1：继续沿用 UpgradeGroup，只加提示

- 优点：改动最小。
- 缺点：隐藏套餐仍会覆盖用户主分组，和需求相冲突。

### 方案 2：为隐藏套餐引入“附加权益订阅”语义

- 优点：只对一类特殊套餐做定向扩展，不影响公开套餐。
- 缺点：需要在订阅消费路径和管理员开通路径增加一层识别逻辑。

### 方案 3：重构整套订阅权限系统

- 优点：长期扩展性最好。
- 缺点：范围过大，不符合本次最小改动目标。

## Chosen Design

采用方案 2。

## Design Details

### 1. 模型同步策略

- 实时读取 CLI API 的模型列表，找出尚未同步到 New API 的：
  - 剩余 Claude 模型
  - Gemini 新模型
- 对已存在模型不重复创建，不覆盖既有渠道和元数据。
- 新模型按既有分组规则追加：
  - Claude -> `claude_core`
  - Gemini -> `gemini_core`
- 价格优先使用项目现有模型定价体系；如模型未命中既有默认值，则写入保守默认值，后续允许管理员手动修正。
- 不调整现有 `gpt52_unlimited`、`AutoGroups`、订阅套餐升级分组。

### 2. 映射核验策略

- 不重建既有品牌模型，只核验这些核心映射是否还保持正确：
  - `gemini-3-flash`
  - `gemini-3-pro`
  - `gemini-3.1-pro`
  - `claude-haiku-4.5`
  - `claude-sonnet-4.5`
  - `claude-opus-4.5`
- 若发现某个渠道的 `model_mapping` 漂移，则只修复异常条目。

### 3. 隐藏套餐的附加权益模型

- 为 `SubscriptionPlan` 增加一个最小识别字段，用于标记“附加权益套餐”。
- 该字段只服务本次隐藏套餐，公开套餐默认关闭。
- 对标记为附加权益的套餐：
  - 创建订阅时不再更新 `user.group`
  - 失效/删除时也不参与主分组回退
- 它仍然是一条正常的 `UserSubscription`，拥有自己的：
  - 总额度
  - 已用额度
  - 每日重置
  - 起止时间

### 4. 消费优先级

- 调用 GPT 系列模型时：
  1. 先尝试消费有效的隐藏 GPT 福利订阅
  2. 若隐藏福利额度不足，再回落到用户当前主套餐或钱包
- 调用非 GPT 系列模型时：
  - 跳过隐藏福利订阅
  - 直接走用户当前主套餐或钱包

这保证：

- 免费用户只要有隐藏福利，也能使用 GPT。
- 付费用户可以把隐藏福利和主套餐叠加起来用。
- 非 GPT 模型不会错误消耗隐藏福利。

### 5. 后台提醒与防重

- 后端新增一个“历史是否领过隐藏套餐”的判定函数。
- 管理员开通隐藏套餐时：
  - 前端先拉用户历史订阅，如果已有该套餐历史记录，显示明显提示。
  - 提交时后端再次校验；若历史已存在，直接拒绝创建。
- 防重以套餐历史记录为准，不因订阅已过期而放宽。

## Data Changes

- `subscription_plans` 增加一个最小布尔字段，类似 `stackable_bonus` 或 `benefit_mode`。
- 仅隐藏套餐会被标记为附加权益。
- 现有普通套餐数据无需迁移结构，只需保持默认值。

## Error Handling

- 同步模型时若某个模型价格没有默认值：
  - 不中断整批同步
  - 记录告警并使用保守默认值
- 管理员重复开通隐藏套餐时：
  - 返回明确业务错误，例如“该用户历史已领取过该新用户福利，不能再次开通”
- 若隐藏福利与主套餐同时存在：
  - 不报冲突，按模型类型和优先级路由扣费

## Testing Strategy

- 后端测试：
  - 隐藏福利订阅创建时不修改用户主分组
  - 历史已领取隐藏套餐时禁止再次开通
  - GPT 请求优先命中隐藏福利
  - 非 GPT 请求不消耗隐藏福利
  - Gemini / Claude 新模型同步后进入正确分组
  - 既有 Gemini 3 / 3.1 与 Claude 4.5 映射保持完好
- 前端测试：
  - 管理员用户订阅管理中，选中隐藏套餐时能显示历史提示
  - 已领过用户的隐藏套餐创建按钮会被限制或提交后得到明确错误
- 集成验证：
  - 模型广场包含新增 Claude/Gemini 模型
  - 调用 GPT 与非 GPT 模型时，扣费路径符合预期

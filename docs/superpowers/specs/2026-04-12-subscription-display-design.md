# 订阅套餐展示配置与商业化套餐页设计

## 背景

当前订阅套餐能力已经能满足基础购买和权益发放，但前台展示仍偏向管理后台风格，缺少商业化定价页常见的层次感和可运营性。现有问题主要集中在三个方面：

1. 用户侧入口名称仍为“钱包管理”，无法准确表达当前页面已以订阅套餐为主。
2. 套餐配置只有业务逻辑字段，运营无法独立维护用于前台展示的标题、标签、卖点、按钮文案等营销内容。
3. 套餐卡片的视觉层次不足，邀请奖励模块与订阅套餐的页面关系也不够清晰。

本次设计目标是在尽量不打散原项目结构的前提下，让套餐页更适合商业化订阅售卖，并保留现有支付、账单、订阅权益等业务链路。

## 目标

- 将用户侧菜单“钱包管理”更名为“套餐与充值”。
- 保持 `/console/topup` 路由和现有支付接口不变。
- 管理端新增“前端展示配置”，支持结构化字段和自由补充文案。
- 用户侧套餐区改造成商业化 pricing grid。
- 桌面端套餐区以 2 到 3 列响应式展示，6 个套餐稳定分成两行。
- 手机端套餐区保持单列。
- 邀请奖励模块移动到订阅套餐区下方展示。
- 保持现有订阅逻辑字段、购买流程、支付流程和邀请奖励逻辑不变。

## 非目标

- 不新增新的支付方式或账单能力。
- 不重构订阅业务主链路。
- 不引入独立活动页或多套运营版本。
- 不改动 `/console/topup` 路由结构。

## 用户体验设计

### 用户侧入口

- 左侧个人区域中的“钱包管理”更名为“套餐与充值”。
- 页面仍然使用当前 `/console/topup` 路由。
- 页面主视图只保留订阅套餐入口，不再恢复额度充值区块。

### 用户侧页面结构

页面按以下顺序展示：

1. 顶部概览卡
   - 保留当前账户/订阅摘要与账单入口。
   - 弱化充值语义，强化订阅语义。
2. 套餐定价区
   - 使用商业化 pricing grid。
   - 桌面端 `xl/lg` 为 3 列，`md` 为 2 列，手机端为 1 列。
3. 邀请奖励区
   - 移动到套餐定价区下方。
   - 作为次级权益模块展示，不抢套餐主视觉。

### 套餐卡片层次

每张卡片按固定层次展示：

1. 眉题 `eyebrow`
2. 高亮标签 `highlight_tag`
3. 展示标题 `display_title`
4. 展示副标题 `display_subtitle`
5. 价格与价格说明 `price_caption`
6. 卖点列表 `feature_points`
7. 真实业务权益摘要
8. 自由补充文案 `display_notes`
9. CTA 按钮 `cta_text`

推荐套餐通过 `is_featured` 和 `theme` 共同控制高亮样式，例如更强边框、浅色背景、强调按钮。

### 展示回退规则

为保证旧套餐可继续使用：

- `display_title` 为空时回退到 `title`
- `display_subtitle` 为空时回退到 `subtitle`
- `cta_text` 为空时回退到“立即订阅”
- `feature_points` 为空时回退到由业务字段生成的权益摘要
- `display_notes` 为空时不展示补充说明区

## 数据模型设计

在订阅套餐模型上新增两类展示字段：

1. `display_config`
   - JSON 对象，用于结构化展示配置
2. `display_notes`
   - 多行文本，用于自由补充文案

建议的 `display_config` 结构：

```json
{
  "display_title": "Plus",
  "display_subtitle": "适合长期高频使用",
  "eyebrow": "推荐选择",
  "highlight_tag": "限时优惠",
  "cta_text": "立即开通",
  "price_caption": "按月自动续费",
  "quota_caption": "每月专属额度",
  "feature_points": [
    "高速响应",
    "更高并发",
    "优先使用核心模型"
  ],
  "theme": "featured",
  "is_featured": true
}
```

`theme` 第一阶段只支持少量枚举：

- `default`
- `featured`
- `dark`

## 管理端设计

`创建新的订阅套餐` / `更新套餐信息` 的 SideSheet 拆成两组：

1. 套餐逻辑配置
   - 现有真实业务字段
2. 前端展示配置
   - 展示标题
   - 展示副标题
   - 眉题
   - 高亮标签
   - CTA 按钮文案
   - 价格说明
   - 额度说明
   - 卖点列表（固定 5 条以内）
   - 自由补充文案
   - 展示主题
   - 是否推荐

运营填写这些字段时，不影响实际权益发放逻辑。

## 邀请奖励模块调整

邀请奖励模块沿用现有逻辑与接口，只调整页面位置和层级：

- 从当前与充值内容并列的结构，调整为位于订阅套餐区下方。
- 视觉上弱于套餐定价区，但仍保留收益统计、邀请链接、划转到余额功能。
- 不更改邀请奖励的计算、划转与复制逻辑。

## 技术边界

### 后端

- 扩展订阅套餐 model，增加 `display_config` 与 `display_notes`
- 管理端订阅套餐创建/更新接口支持读写这两个字段
- 订阅套餐列表接口返回这两个字段
- 旧数据兼容：无展示配置时返回空值，由前端回退

### 前端

- 修改侧边栏文案
- 扩展套餐管理表单
- 重做订阅卡片渲染
- 调整邀请奖励模块位置
- 保持账单入口、购买弹窗、支付流程不变

## 风险与控制

### 风险 1：旧套餐数据没有展示配置

处理方式：

- 后端允许字段为空
- 前端统一做回退渲染

### 风险 2：卖点字段输入不规范导致卡片溢出

处理方式：

- 结构化卖点限制为最多 5 条
- 自由补充文案最多展示 5 行
- 管理端提供明确占位说明

### 风险 3：卡片风格过强，和现有项目风格冲突

处理方式：

- 维持现有 Semi UI 基础组件体系
- 通过卡片层次、间距、边框、背景和按钮强化商业化视觉，而不是引入全新设计语言

## 测试策略

### 后端

- 新增展示字段的序列化/反序列化测试
- 创建套餐时展示字段落库测试
- 更新套餐时展示字段更新测试
- 旧数据兼容测试

### 前端

- 展示字段回退逻辑测试
- 卖点列表与自由补充文案渲染测试
- 不同周期下额度标签继续正确展示
- 套餐卡片桌面端/移动端响应式类名验证
- 邀请奖励模块移动后仍可正常操作

## 预期结果

完成后，系统应具备以下表现：

- 用户从左侧菜单看到“套餐与充值”
- 进入页面后，先看到商业化订阅套餐卡片，再看到邀请奖励
- 管理员在创建套餐时，既能配置真实业务逻辑，也能配置前台展示内容
- 6 个套餐在桌面端稳定呈现为两行 pricing grid，手机端单列展示
- 旧套餐在不补展示配置的情况下仍能正常渲染与购买
